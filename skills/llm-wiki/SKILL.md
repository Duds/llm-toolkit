Atom-first knowledge base management. Atoms are the source of truth; wiki pages are compiled views. Raw sources are immutable. Every claim cites its source.

## Setup

Before any sub-command runs:

1. Resolve the wiki root via the loader script.
2. Read the returned JSON in full. Don't pipe through grep, head, or jq.
3. **Load the sub-command's reference file.** Bootstrap without `bootstrap.md` loaded skips the scenario-template selection. Ingest without `ingest.md` skips the two-step chain-of-thought.

Skipping these produces atoms with no provenance, wiki pages that drift from atoms, or compiled output the user can't trace.

### 1. Wiki root resolution

```bash
node .claude/skills/llm-wiki/scripts/load-wiki.mjs
```

The loader walks up from cwd looking for `_llm-wiki/`, then falls back to `~/Projects/_llm-wiki/` (portfolio-level). It returns JSON:

```json
{
  "found": true,
  "wikiRoot": "absolute path",
  "level": "project" | "portfolio",
  "hasPurpose": true,
  "purpose": "purpose.md contents",
  "hasSchema": true,
  "schema": "schema.md contents",
  "atomCount": 0,
  "wikiPageCount": 0
}
```

If `found: false` and the user asked for anything other than `bootstrap`, stop and suggest `/llm-wiki bootstrap`. The other commands have no wiki to operate on.

If `hasPurpose: false`, nudge once: *"Run `/llm-wiki bootstrap --scenario <scenario>` or write `purpose.md` manually — purpose is the wiki's directional intent and every ingest reads it."* Then proceed.

Override the resolved root with `LLM_WIKI_ROOT=path/to/wiki` (absolute or cwd-relative).

If the loader output is already in this session's history, don't re-run. Exceptions: you just ran `/llm-wiki bootstrap` (it created the wiki), or the user manually edited `purpose.md` / `schema.md`.

### 2. Scope

Every wiki operation is at one of two scopes, identified once and cached for the session:

- **Portfolio** (`~/Projects/_llm-wiki/`): cross-project knowledge, shared patterns, reference material.
- **Project** (`~/Projects/<project>/_llm-wiki/`): project-specific decisions, architecture, research.

The loader returns `level`. The two scopes coexist — portfolio wikis link to project wikis. Same skill, same commands, same atom schema.

## Shared laws

Apply to every sub-command. Match-and-refuse: if you're about to violate one, stop and route through the correct command.

### Atoms are the source of truth

- Edit atoms, not wiki pages. `wiki/` is derived — running `/llm-wiki compile` regenerates it from atoms.
- Atom IDs are immutable. Format: `atom-YYYYMMDD-NNN`. Once written, the ID never changes.
- Superseded atoms stay in place with `status: superseded` and `superseded-by: atom-newer-id`. Never delete or overwrite.
- Confidence is a number, not a label. Score 0.0–1.0. See [reference/atoms.md](reference/atoms.md).

### Raw is immutable

- Never edit files in `raw/` after ingest. Process them into atoms instead.
- Every raw file gets a row in `raw/_provenance.md` at ingest time.
- If a raw source changes, copy the new version with a date suffix; don't overwrite.

### Every claim cites

- Atoms without at least one `sources:` entry are invalid. `lint` flags them as critical.
- The single exception: `branch: meta` atoms (wiki structure, conventions) can omit sources.

### Wikilinks are typed

- `[[atom-YYYYMMDD-NNN]]` → links to an atom file
- `[[wiki/branch/page-name]]` → links to a compiled wiki page
- `[[raw/source-filename]]` → links to a raw source

Plain markdown links to wiki content (`[text](path)`) are wrong. Use wikilinks. Obsidian needs them; the LLM needs them; humans need them.

### Compile, don't hand-write

- `wiki/` files are generated by `/llm-wiki compile`. Never hand-edit them.
- The `_README.md` at the top of `wiki/` makes this explicit; the compiler refuses to overwrite a file lacking the auto-generated header without `--force`.

### Absolute bans

- **Atoms with no source.** Either cite a `raw/` file or set `branch: meta`. Never both blank.
- **Editing wiki pages directly.** They are derived. Run compile.
- **Skipping the two-step ingest.** Single-pass extraction loses contradictions and over-collapses. See [reference/ingest.md](reference/ingest.md).
- **Inventing atom IDs.** IDs come from the timestamp scheme. Never reuse an ID.
- **Cross-wiki atom IDs without prefix.** Portfolio wiki referencing a project atom uses `[[project/atom-id]]`, not bare `[[atom-id]]`.

### Routing rules

1. **No argument**: render the commands table grouped by category. Ask what they'd like to do.
2. **First word matches a command**: load its reference, run its instructions. Everything after the command name is the target.
3. **First word doesn't match**: treat as a general wiki query. Run setup, then `/llm-wiki query <full argument>`.

Setup runs once per session. Sub-commands don't re-invoke `/llm-wiki`.

If the wiki doesn't exist and the user asks for anything other than `bootstrap`, stop. Don't fake a wiki root.

## Pin / Unpin

`pin` creates a standalone shortcut so `/<command>` invokes `/llm-wiki <command>` directly. `unpin` removes it. The script writes to every harness directory that has llm-wiki installed.

```bash
node .claude/skills/llm-wiki/scripts/pin.mjs <pin|unpin> <command>
```

Valid commands: `bootstrap`, `crawl`, `ingest`, `compile`, `query`, `lint`, `report`.

Recommended pins: `query` (daily use), `ingest` (frequent), `lint` (pre-commit). Leave `bootstrap` unpinned — it's a once-per-wiki action.
