# Purpose.md — directional intent of the wiki

Foundational reference. Loaded by `bootstrap` (writes the initial file) and read by `ingest`, `compile`, and `query` on every invocation. Defines what `purpose.md` is, why it exists, and how to write or update it.

## Purpose vs schema

A wiki has two configuration files. They look similar at first; they're different jobs.

| File | Role | Changes when |
|---|---|---|
| `schema.md` | **Structural rules** — entity types, frontmatter shape, branch list, lint rules | Schema of the wiki itself changes (rare) |
| `purpose.md` | **Directional intent** — goals, key questions, scope, evolving thesis | Wiki's focus shifts, new questions emerge, thesis updates |

`schema.md` answers *"how is this wiki shaped?"*. `purpose.md` answers *"what is this wiki for, and what do I want to learn from it?"*

Both must exist. Bootstrap writes both. Every other command reads both.

## Why purpose.md matters

Atoms generated by `ingest` and `crawl` are biased by what the LLM thinks is interesting. Without `purpose.md`, the LLM defaults to generic salience: who, what, when, where. With `purpose.md` loaded, the LLM filters for relevance to *this wiki's* questions.

Concretely: a DCCEEW ICT wiki without purpose.md will surface generic atoms about the department's structure. The same source ingested with a purpose.md that says *"focus on legacy system dependencies, FY2027 sunset risk, and statutory function coverage"* will surface atoms about PAS, EIAS, and EPBC Act section coverage — the things that matter.

This is the single biggest quality lever for the wiki. Treat `purpose.md` as load-bearing.

## Structure

```markdown
---
title: "Purpose"
last-updated: YYYY-MM-DD
---

# Purpose

## What this wiki is for

{{One paragraph. Who's the wiki for, what decisions does it support, what's the scope?}}

## Key questions

{{3-7 questions the wiki should help answer. Specific. Action-relevant.}}

1. {{Question one}}
2. {{Question two}}
3. ...

## Evolving thesis

{{The current working hypothesis or position. What does the wiki currently believe? This section updates as the wiki learns.}}

## Scope

**In scope:**
- {{Topic / domain / time period}}
- ...

**Out of scope:**
- {{Topic the wiki deliberately does not cover}}
- ...

## Anti-references

{{What this wiki is NOT. Helps the LLM avoid drift. Optional but valuable.}}

- Not a {{X}}
- Not for {{Y}}
```

## Scenario templates

`bootstrap --scenario <name>` preconfigures `purpose.md` for common cases:

| Scenario | Typical "what this is for" | Key questions tend to be |
|---|---|---|
| `research` | Synthesise a body of literature on a topic | What are the dominant positions? Where do they disagree? What's missing? |
| `consulting` | Capture client context, recommendations, lessons | What does the client need? What constraints bind us? What worked / didn't? |
| `design` | Service or system design knowledge base | What are the user needs? What are the touchpoints? Where are the gaps? |
| `general` | Catch-all personal knowledge base | (Left for you to fill in) |

Pick the closest scenario at bootstrap time. Edit afterwards to match the actual wiki's needs.

## How sub-commands use purpose.md

- **`ingest`**: passed to the LLM in the analysis step (Step 1 of the two-step ingest) as filter context. Atoms unrelated to purpose questions get lower confidence; atoms directly answering a purpose question get higher confidence.
- **`crawl`**: used to prioritise candidate sources. A source that looks relevant to the purpose questions ranks higher in the review queue.
- **`compile`**: groups atoms by which purpose question they help answer. Wiki pages can be structured around questions instead of branches.
- **`query`**: combined with the user's question to ground synthesis. If the user's question is well-aligned with a purpose question, prior atoms about it are prioritised.

If `purpose.md` is missing or empty (placeholder TODO markers), the LLM falls back to generic salience. Sub-commands warn once per session: *"purpose.md is empty — atom quality will suffer. Edit it or run bootstrap again."*

## Updating purpose.md

Purpose evolves. The wiki teaches you what to ask. When the thesis section is stale or the key questions don't match what you actually use the wiki for:

1. Edit `purpose.md` directly.
2. Bump `last-updated:` to today.
3. Consider running `/llm-wiki report` to see if the existing atom confidence distribution still matches the new purpose.

Don't track purpose history in the file itself — that's what git is for. Keep purpose.md current.

## Authoring rules

- **Concrete questions, not topics.** "Where is statutory coverage incomplete?" beats "EPBC Act compliance."
- **Action-relevant.** Each key question should be one Dale would actually ask. If you can't imagine asking it, drop it.
- **3–7 questions.** Fewer makes the wiki narrow; more makes purpose.md meaningless as a filter.
- **Update the thesis when it changes.** A stale thesis is worse than no thesis — the LLM will filter against an out-of-date view of the world.
- **Anti-references are a force multiplier.** Saying what the wiki *isn't* prevents drift more cheaply than saying what it is. Two or three anti-references go a long way.

## Anti-patterns

- **Purpose.md as a project plan.** Not the place. The wiki captures knowledge; the project plan lives elsewhere (TASKS.md, sprint plans).
- **Purpose.md as a summary of contents.** That's `overview.md`, auto-generated by `compile`. Purpose is forward-looking intent, not retrospective summary.
- **Editing purpose.md to match the atoms.** Backwards. Purpose drives ingest; ingest drives atoms. If atoms don't match purpose, that's signal — either purpose has shifted or ingest is missing things.
- **Generic key questions copied from the scenario template.** Replace them with Dale's actual questions before the first ingest.
